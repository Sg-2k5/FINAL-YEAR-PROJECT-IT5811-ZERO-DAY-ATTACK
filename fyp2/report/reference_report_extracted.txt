ABSTRACT
Healthcare institutions possess valuable patient data for training artificial intelligence
models, but privacy regulations, data sovereignty concerns, and security requirements
prevent centralized data aggregation. Federated Learning enables collaborative model
training without raw data sharing, but existing approaches lack comprehensive trust
evaluation and security mechanisms essential for healthcare applications. This project
develops aTrust-Aware Vertical Federated Learning Frameworkthat addresses
these challenges through multi-dimensional trust scoring, anomaly detection, and real-
time monitoring.
The framework integrates three healthcare clients with distinct data modalities:
blood glucose measurements for diabetes classification using neural networks, retinal
fundus images for diabetic retinopathy detection using ResNet18 CNN, and medication
review sequences for drug effectiveness prediction using bidirectional LSTM. A central
server coordinates federated training using the Flower framework, implementing trust-
weighted aggregation that evaluates client reliability across five factors — accuracy
stability (30%), anomaly behavior (25%), participation consistency (20%), historical
reputation (15%), and resource usage patterns (10%). Clients with trust scores below
the threshold (τ= 0.6) are excluded from aggregation, protecting the global model
from unreliable or malicious participants.
Experimental evaluation on standard healthcare datasets (Pima Indians Diabetes,
APTOS Retinal Images, and UCI Drug Reviews) demonstrates that trust-weighted
aggregation outperforms standard FedAvg by 0.8–1.7% accuracy across tasks, while
successfully mitigating gradient poisoning, Sybil attacks, and Byzantine failures.
The framework achieves 79.3% accuracy for blood glucose prediction, 79.6% for
retinal classification, and 72.4% for medication effectiveness prediction. This work
demonstrates that secure, trustworthy, and compliant collaborative AI is achievable
in healthcare environments through comprehensive trust evaluation and multi-layered
security, enabling multi-institutional learning while preserving patient privacy.

Chapter 1
INTRODUCTION
1.1 BACKGROUND AND MOTIV ATION
Healthcare institutions possess vast patient data valuable for machine learning but
face barriers due to privacy regulations (HIPAA, GDPR), data sovereignty concerns,
and heterogeneous data formats. Traditional centralized machine learning requires
consolidating data in one location, which is untenable in healthcare due to privacy,
legal, and ethical constraints.
Federated Learning (FL) enables collaborative model training across distributed
data sources without raw data centralization. By training models locally and sharing
only model parameters, FL preserves privacy while leveraging collective knowledge.
This approach improves model accuracy through diverse patient populations, enhances
privacy protection, ensures regulatory compliance, and accelerates clinical deployment
of AI technologies.
1.2 CHALLENGES
While federated learning offers promising solutions for privacy-preserving collabora-
tive healthcare AI, several critical challenges must be addressed to ensure practical
deployment and trustworthy operation in real-world clinical environments.
1.2.1 Trust and Reliability Issues
Participating institutions exhibit heterogeneous characteristics: data quality variations,
computational capabilities, and network reliability. Without trust evaluation, all
1

===== PAGE 13 =====

clients are treated equally regardless of data quality. This causes problems when
unreliable clients with poor data or inconsistent participation degrade the global model.
Traditional federated averaging lacks defense against malicious participants who inject
poisoned updates.
1.2.2 Security Threats
Federated learning faces multiple attack vectors: Gradient Poisoning (manipulated
gradients degrade model), Model Inversion (reconstructing training data from pa-
rameters), Byzantine Attacks (malformed updates disrupt aggregation), Sybil Attacks
(fake client identities amplify malicious influence), and Reconnaissance (probing for
vulnerabilities).
1.2.3 Lack of Accountability
Existing implementations lack comprehensive audit mechanisms for tracking client
behaviors and providing verifiable evidence. Without immutable audit trails, determin-
ing responsibility for incidents becomes impossible. HIPAA mandates detailed audit
logging, but standard FL frameworks provide minimal capabilities. This creates moral
hazard where participants face no consequences for poor behavior.
1.3 PROBLEM STATEMENT
Existing federated learning methods lack advanced mechanisms to reliably evaluate
client trust, leaving systems vulnerable to attacks like gradient poisoning, Byzantine
faults, and Sybil infiltrations. They also miss immutable audit trails needed for
regulatory compliance and do not fully support vertical federated learning with diverse
data types. Addressing these gaps is crucial for secure, trustworthy collaborative AI in
sensitive domains.

This project develops a Trust-Aware Vertical Federated Learning Framework

1.4 PROPOSED SOLUTION
The Trust-Aware Vertical Federated Learning Framework integrates multiple comple-
mentary mechanisms to address the challenges identified above, creating a comprehen-
sive system for secure, trustworthy, and accountable collaborative healthcare AI.
1.4.1 Trust-Based Client Weighting
The framework evaluates client reliability across five dimensions: Accuracy Stability
(30%), Anomaly Behavior (25%), Participation Consistency (20%), Historical Repu-
tation (15%), and Resource Usage (10%). The composite trust score determines each
client’s weight in aggregation, ensuring reliable participants dominate while unreliable
clients are down-weighted or excluded.
1.4.2 Multi-Layered Security Mechanisms
The framework implements: Statistical Anomaly Detection (z-score analysis for
time anomalies), ML-Based Update Validation (Isolation Forest for outlier detection),
Behavioral Monitoring (CUSUM charts for drift detection), Honeypot Deception (trap
reconnaissance attempts), and Automatic Quarantine (exclude malicious clients).
1.4.3 Blockchain-Based Audit Trail
Ethereum smart contracts create immutable records of all FL activities (training rounds,
client submissions, trust evaluations, security incidents). Events are cryptographically
hashed and permanently stored, providing tamper-proof audit trails for HIPAA

1.5 OBJECTIVES
This project aims to implement federated learning infrastructure using Flower frame-
work for three healthcare clients handling blood, retinal, and medication data. The
framework develops a five-factor trust evaluation mechanism for client reliability as-
sessment and integrates multi-layered anomaly detection using z-score, Isolation Forest,

poisoning, Byzantine, and Sybil attacks. A Streamlit-based administrative dashboard

zero raw data transfer. The project evaluates model accuracy, security robustness, and
computational efficiency across distributed healthcare environments.
1.6 SCOPE OF THE PROJECT
In Scope: Vertical FL with three healthcare clients (blood/retinal/medication), trust-
weighted aggregation, multi-layered security, blockchain audit trails, deep learning
models (NN, CNN, LSTM), Streamlit dashboard, evaluation on Pima/APTOS/UCI

Chapter 2
LITERATURE SURVEY
2.1 FEDERATED LEARNING IN HEALTHCARE
FedAvg [1] averages model weights across clients but assumes equal trustworthiness
and lacks defense against malicious behavior. The framework implements simple
averaging without considering client reliability, making it vulnerable to poisoning
attacks. Recent surveys on FL advances [2] did not propose concrete trust mechanisms
for vertical FL scenarios where different institutions hold complementary feature
spaces. Applications of FL to medical imaging [3][7] focused only on horizontal FL
with identical feature spaces, limiting applicability to real-world healthcare scenarios
where data modalities differ across institutions.
2.2 TRUST AND SECURITY IN FEDERATED SYSTEMS
Reputation-based FL approaches [4] rely solely on loss values, missing critical
behavioral patterns such as participation consistency and resource usage anomalies.
Client selection mechanisms lack explicit trust modeling across multiple dimensions.
These approaches fail to address the multi-dimensional nature of trust in healthcare
federated learning environments.
2.3 VERTICAL FEDERATED LEARNING
VFL taxonomy [5] with SecureBoost using homomorphic encryption lacks trust
assessment mechanisms for heterogeneous participants. Split learning approaches for
vertical data partitioning assume all participants are trustworthy, creating vulnerabilities

in adversarial environments. Differential privacy additions [2] to VFL reduce model
accuracy significantly and don’t address trust heterogeneity across clients with different
data modalities and model architectures.
2.4 ANOMALY DETECTION IN DISTRIBUTED SYSTEMS
Krum [6] for Byzantine-robust aggregation requires high computational complexity
withO(n 2)pairwise distance calculations. Coordinate-wise median and trimmed
mean rules [6] may fail against sophisticated adaptive attacks that target statistical
thresholds. Poisoning detection using validation data requires server-side labeled
data unavailable in privacy-preserving healthcare scenarios where data cannot leave
institutional boundaries.
2.5 PRIV ACY-PRESERVING TECHNIQUES
Applications of FL to brain tumor segmentation [7] lack trust mechanisms to handle
unreliable medical imaging from diverse scanner types. Hybrid FL approaches for
Electronic Health Records [3] assume all participating hospitals are trusted entities
without verification mechanisms. Surveys on privacy-preserving machine learning
[2] note the fundamental trade-off between privacy guarantees through differential
privacy and model accuracy, particularly severe in medical diagnosis tasks requiring
high precision.
2.6 MULTI-MODAL MEDICAL DATA INTEGRATION
Research on integrating heterogeneous medical data modalities [3][5] remains limited.
Blood biomarkers, retinal imaging, and medication histories provide complementary
diagnostic information for diabetes prediction, yet existing FL frameworks lack
architectures supporting simultaneous training of distinct model types (neural networks

===== PAGE 19 =====

for tabular data, CNNs for imaging, LSTMs for temporal sequences) while maintaining
trust evaluation across these diverse client types.
2.7 SUMMARY OF EXISTING WORK
Literature reveals critical gaps: (1) trust mechanisms focus on accuracy-based repu-
tation, neglecting behavioral patterns, resource usage, and participation consistency
[1][4], (2) frameworks designed for horizontal FL lack vertical FL trust assessment
across heterogeneous data modalities [5], (3) systems lack real-time monitoring
dashboards with interpretable trust visualizations, (4) anomaly detection relies on sta-
tistical methods vulnerable to adaptive attacks [6], (5) few address healthcare-specific
requirements including HIPAA compliance, comprehensive audit trails, multi-modal
data integration, and clinical interpretability [3][7], and (6) existing VFL approaches
assume homogeneous client models rather than supporting diverse architectures tailored
to different data types.
2.8 RESEARCH OBJECTIVES
This project aims to: (1) develop multi-factor trust evaluation across five dimensions
with weighted scoring (accuracy stability 30%, anomaly detection 25%, participation
consistency 20%, historical behavior 15%, resource usage 10%) [1][2], (2) design trust-
weighted aggregation for vertical FL supporting heterogeneous client architectures [5],
(3) create real-time monitoring dashboard with interpretable trust metrics, (4) integrate
multi-layered security combining Isolation Forest anomaly detection with statistical
z-score analysis [6], and (5) evaluate on healthcare diabetes datasets with blood
reports, retinal scans, and medication histories against adversarial attacks including data
poisoning, model poisoning, and free-riding behaviors [3][7].

2.9 PROPOSED SOLUTION ARCHITECTURE
The Trust-Aware Vertical FL Framework comprises five core modules implementing
comprehensive trust-based federated learning for healthcare applications.
2.9.1 Client Management Module
Manages three healthcare client types with distinct data modalities: (1) Blood Re-
ports Client processing structured medical data (glucose, HbA1c, cholesterol, BMI,
insulin, blood pressure) using Logistic Regression and Multi-Layer Perceptron neural
networks with PyTorch implementation featuring 64-32-16 hidden layers with ReLU
activation and dropout regularization [1], (2) Retinal Scans Client handling image
data through Convolutional Neural Networks for diabetic retinopathy detection with
gradient compression for efficiency [3][7], and (3) Medication History Client analyzing
sequential medication data (drug sequences, dosage, duration, response patterns) using
Long Short-Term Memory networks for temporal pattern recognition. Each client
implements local model training with differential privacy guarantees [2] and secure
encrypted communication with the federated server.
2.9.2 Federated Learning Server Module
Coordinates entire federated learning process through Core Coordination Engine
managing round scheduling, trust-based client selection, global model distribution, and
FL protocol compliance enforcement [1][4]. Communication Handler provides secure
message brokering with update validation and heartbeat monitoring. Model Manage-
ment maintains global model repository with versioning, checkpoint persistence, and
convergence tracking across heterogeneous client architectures [5].

2.9.3 Trust Evaluation Engine Module
Dynamically assesses client trustworthiness using five-factor scoring formula:
T rustscore = 0.30×Accuracy stability + 0.25×Anomaly score + 0.20×
P articipationconsistency + 0.15×Historical behavior + 0.10×Resource ef f iciency
with configurable threshold of 0.6 [4]. Implements real-time trust score computation
through Metric Calculators tracking model performance consistency, suspicious
behavior identification, engagement patterns, long-term behavior analysis, and system
usage efficiency.
2.9.4 Secure Aggregation Module
Performs trust-weighted model update combination through Trust-Weighted Aggrega-
tor calculating contribution weights based on trust scores, validating update integrity,
and fusing multi-modal outputs [1][5]. Byzantine Defense System implements trimmed
mean aggregation for outlier resistance, coordinate-wise median robust aggregation,
and Krum algorithm for Byzantine-robust selection [6]. Privacy-Preserving Aggrega-
tion employs differential privacy with formal guarantees [2] ensuring clients with trust
scores below 0.6 are excluded from parameter aggregation.
2.9.5 Admin Dashboard and Monitoring Module
Real-time monitoring panel built with Streamlit provides administrative interface
featuring trust score evolution visualization, live training progress metrics, security
threat alerts, and performance tracking. Control Interface enables manual intervention
capabilities, runtime parameter adjustment, and client lifecycle management. Analytics
Engine generates automated performance reports, comparative analysis against baseline
FedAvg [1], and research-quality visualizations for clinical stakeholders.

3.1 SYSTEM ARCHITECTURE
The framework employs a distributed client-server architecture with three layers: Edge
Client, Central Server, and Security & Monitoring. Figure 3.1 illustrates the system
architecture.
The architecture is designed to support heterogeneous healthcare institutions by
allowing each edge client type (Blood, Retinal, Medication) to run its own data pipeline
and model while exposing a unified Flower NumPyClient interface towards the central
server. This decoupling enables each hospital to independently maintain and upgrade
its local models and preprocessing without requiring any change to the orchestration
logic, which only interacts with standardized client methods for parameter exchange
and training control.
The Security & Monitoring layer is logically separated from the core federated
learning orchestration to avoid tight coupling between training logic and security
policies. Anomaly detectors and monitoring components subscribe to telemetry streams
(training times, model update statistics, participation logs, resource usage) and feed
their outputs into the trust engine and dashboard, allowing security mechanisms to
evolve (e.g., new detectors or thresholds) without modifying the main aggregation and
coordination flow.

4.9 DEPLOYMENT ARCHITECTURE
4.9.1 Component Deployment
The deployment architecture for the trust-aware federated learning system is organized
into three primary zones: Edge Processing, Fusion Layer and Server, and Security and
Audit. At the edge, diverse healthcare data sources like diabetes blood reports, retinal
images, and medication histories are locally preprocessed and used to train specialized
models. Each edge client implements behavior monitoring, local model training, and
generates updates that are subject to real-time anomaly analysis. The Client Side
Anomaly Monitor analyzes training time, accuracy consistency, resource usage, and
flags suspicious activity before allowing updates to enter the federation process.
The central Fusion Layer and Server orchestrate collaborative training with modules
for aggregation, filtering, recovery, and trust evaluation. Here, low trust clients are
excluded, and model outputs from verified participants are combined using weighted
averaging, fusion, or meta-classifier stacking. Trust scores and anomaly patterns
detected at the edge help filter unreliable contributions. The server coordinates
global training sessions and logs performance, client profiles, and round results for
transparency. Refinement recovery protocols ensure excluded clients can be safely
reinstated after review, supporting resilience and adaptive learning.
Security and audit integration in the architecture uses a blockchain ledger to ensure
immutable and transparent audit history. Client health monitoring detects and isolates
offline or malicious clients through honeypots and quarantine mechanisms. Real-time
admin reports provide trust visualizations and isolated audits, delivering robust security,
full traceability, and adaptive learning critical for healthcare data protection.

CONCLUSION AND FUTURE WORK
6.1 CONCLUSION
This project successfully developed a Trust-Aware Vertical Federated Learning Frame-
work for collaborative healthcare diabetes prediction, integrating multi-dimensional
trust evaluation and security mechanisms. The framework demonstrates significant
improvements in federated learning reliability through trust-weighted aggregation using
five factors, achieving accuracy improvements over traditional FedAvg approaches
while maintaining robust security with high detection rates and minimal false positives.
The implemented vertical FL architecture enables three healthcare institutions to
collaboratively train specialized models without sharing raw data, addressing critical
privacy concerns in medical AI applications. Experimental validation across blood
reports, retinal scans, and medication history data demonstrates strong predictive
accuracy for diabetes detection. The Flower-based system with comprehensive trust
scoring, multi-technique anomaly detection, and Streamlit dashboard for real-time
monitoring successfully achieves HIPAA compliance while maintaining computational
efficiency with reasonable overhead and sub-linear scalability.
Limitations include public dataset usage potentially not representing clinical diver-
sity, controlled environment testing not reflecting real-world connectivity challenges,
communication overhead constraining bandwidth-limited scenarios, and class imbal-
ance affecting rare case performance.
